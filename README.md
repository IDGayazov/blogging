1. Общие сведения
   Приложение представляет собой платформу для создания и публикации блогов. Пользователи могут создавать статьи, редактировать их, комментировать и просматривать контент, создаваемый другими. Система должна поддерживать авторизацию пользователей, организацию контента по категориям, возможность добавления комментариев и взаимодействия с другими пользователями через лайки.

2. Цели и задачи
   Цель проекта — создать веб-приложение для блога, которое позволит пользователям:
   Публиковать, редактировать и удалять статьи.
   Добавлять комментарии к статьям.
   Просматривать статьи по категориям и тегам.
   Управлять своим профилем и подписками.
   Реализовать функциональность авторизации и аутентификации пользователей.
3. Технологии
   Frontend: React.js / Vue.js / Angular
   Backend: Node.js (Express.js) / Django / Flask
   База данных: PostgreSQL / MySQL / SQLite
   Авторизация: JWT (JSON Web Tokens) или OAuth
   Стилизация: CSS, SCSS или CSS-фреймворки (например, Tailwind CSS)
   Хостинг: Heroku / DigitalOcean / AWS
4. Функциональные требования 
- 4.1. Регистрация и авторизация
   Пользователи должны иметь возможность регистрироваться с помощью электронной почты и пароля.
   Вход в систему должен осуществляться по электронной почте и паролю.
   Реализовать восстановление пароля через электронную почту.
   Авторизация должна быть защищена JWT.
- 4.2. Профиль пользователя
   Пользователь должен иметь возможность создать и редактировать свой профиль (имя, фото профиля, краткая биография).
   Возможность смены пароля.
   История публикаций.
- 4.3. Управление статьями
   Пользователи должны иметь возможность создавать, редактировать и удалять свои статьи.
   Статья должна содержать заголовок, текст, дату создания, категорию и теги.
   Статьи должны отображаться на главной странице блога с возможностью фильтрации по категориям и тегам.
- 4.4. Комментарии
   Пользователи могут оставлять комментарии к статьям.
   Комментарии должны поддерживать функциональность ответа на другой комментарий (nested comments).
   Пользователи могут удалять или редактировать свои комментарии. 
- 4.5. Лайки и оценки
   Возможность ставить лайки статьям и комментариям.
   Лайки можно отменить (поставить/снять).
- 4.6. Поиск
 Возможность поиска статей по ключевым словам в заголовках и тексте.
- 4.7. Категории и теги
 Статьи должны быть организованы по категориям и тегам.
 Пользователь должен иметь возможность просматривать все статьи в определённой категории или с определённым тегом.
5. Нефункциональные требования
   Высокая производительность при больших объемах данных.
   Адаптивность интерфейса для мобильных устройств.
   Защищенность данных (например, защита от SQL-инъекций, XSS-атак).
   Оперативное хранение и обработка данных.
6. Описание базы данных
   Основные сущности:
   User (Пользователь):
- id (PK): уникальный идентификатор пользователя.
- email: уникальный адрес электронной почты.
- password_hash: зашифрованный пароль.
- name: имя пользователя.
- avatar_url: ссылка на изображение профиля (опционально).
- bio: краткая биография пользователя (опционально).
- created_at: дата регистрации пользователя.
- updated_at: дата последнего обновления профиля.

Article (Статья):
- id (PK): уникальный идентификатор статьи.
- user_id (FK): внешний ключ на таблицу User.
- title: заголовок статьи.
- content: содержание статьи.
- created_at: дата публикации статьи.
- updated_at: дата последнего изменения статьи.
- category_id (FK): внешний ключ на таблицу Category (если статьи имеют категории).
- tags: список тегов, разделенных запятой.

Comment (Комментарий):
- id (PK): уникальный идентификатор комментария.
- article_id (FK): внешний ключ на таблицу Article.
- user_id (FK): внешний ключ на таблицу User.
- parent_comment_id (FK, NULL): внешний ключ на сам комментарий (для реализации вложенности).
- content: текст комментария.
- created_at: дата публикации комментария.
- updated_at: дата последнего изменения комментария.

Like (Лайк):
- id (PK): уникальный идентификатор лайка.
- user_id (FK): внешний ключ на таблицу User.
- article_id (FK, NULL): внешний ключ на таблицу Article (для лайков статей).
- comment_id (FK, NULL): внешний ключ на таблицу Comment (для лайков комментариев).
- created_at: дата постановки лайка.

Category (Категория):
- id (PK): уникальный идентификатор категории.
- name: название категории.
- description: описание категории (опционально).
- created_at: дата создания категории.

Tag (Тег):
- id (PK): уникальный идентификатор тега.
- name: название тега.
- created_at: дата создания тега.

Article_Tags (Связь статьи с тегами):
- article_id (FK): внешний ключ на таблицу Article.
- tag_id (FK): внешний ключ на таблицу Tag.

Связи между таблицами:

User - Article: один ко многим (один пользователь может создать несколько статей).

User - Comment: один ко многим (один пользователь может оставлять несколько комментариев).

Article - Comment: один ко многим (одна статья может иметь множество комментариев).

Article - Category: многие к одному (множество статей могут быть отнесены к одной категории).

Article - Tag: многие ко многим (множество статей могут иметь множество тегов).

Интеграция с БД:

- Тестирование:

- Проведение тестирования функциональности приложения.

Развертывание и хостинг:
